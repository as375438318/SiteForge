## 1. 背景与问题陈述（Problem Statement）

### 1.1 用户问题

中小企业在官网建设与获客上面临"**四高一低**"困境：

- **设计门槛高**：企业主/市场人员缺乏设计能力，传统建站工具"选模板 → 改内容"仍需大量试错，平均建站周期 3-7 天。
- **找参考成本高**：企业通常知道"我想要类似某某同行那样的网站"，但无法把"参考意图"转化为可编辑结构。
- **建好了获客难**：绝大多数中小企业官网上线后"无人问津"，缺乏 SEO 优化能力，搜索引擎排名靠后；而 2025 年起，**用户搜索行为正从传统搜索引擎向 AI 搜索（ChatGPT、Perplexity、文心一言、豆包、Kimi）迁移**，企业内容若不被 AI 搜索引用，等于在新渠道里"隐形"。
- **数据自主可控诉求高**：越来越多企业不愿把官网数据、客户线索托管在第三方 SaaS 平台——一方面担心平台涨价/停服锁定，另一方面线索数据属于核心资产，希望存在自己服务器上。SaaS 建站无法满足这一诉求。
- **自主可控低**：已上线网站的内容更新依赖技术人员，市场人员无法自主发布。

### 1.2 核心洞察

> **洞察一：企业建站的真实起点不是"空白画布"，而是"我看过同行的网站，我想要类似的结构"。**
>
> **洞察二：建站只是起点，"被搜到、被 AI 引用"才是企业官网的真正价值。一个没人能搜到的官网，等于没有官网。**

当前主流建站产品（凡科、上线了、Wix、Strikingly）均为 **SaaS 形态**，存在三重缺陷：① 数据托管在平台，企业无法自主可控；② 重"建站"轻"获客"，SEO 能力薄弱，几乎无 GEO（AI 搜索优化）能力；③ 模板是平台预设的、有限的、与用户行业不一定匹配的。

本产品以 **"本地部署 + AI 结构复刻 + SEO/GEO 双引擎"** 三位一体切入，解决"建得快、自己控、搜得到、AI 引得到"四个核心问题。

### 1.3 为什么是本地部署

| 维度 | SaaS 建站 | 本地部署（本产品） |
|------|-----------|-------------------|
| 数据归属 | 数据存平台，存在锁定与停服风险 | **数据 100% 存客户自己服务器** |
| 线索资产 | 客户线索进平台数据库 | **线索进客户自有数据库** |
| 定价 | 按年/月订阅，长期成本不可控 | **License 买断 + 年度服务费，长期成本可控** |
| 定制自由度 | 受平台限制 | 可二次开发（源码授权模式） |
| 合规 | 跨境 SaaS 有数据出境问题 | **数据不出企业，天然满足数据合规** |
| 部署门槛 | 零部署 | 需基础 IT 人员一次性部署（Docker 一键化降低门槛） |

### 1.4 为什么 SEO + GEO 缺一不可

- **SEO（搜索引擎优化）**：面向 Google、百度、Bing 等传统搜索引擎，优化技术结构与内容，提升自然排名。仍是当前企业获客主渠道。
- **GEO（Generative Engine Optimization，生成式引擎优化）**：面向 ChatGPT、Perplexity、文心一言、豆包、Kimi 等 AI 搜索引擎，优化内容的**可引用性**与**权威性**，让企业内容更容易被 AI 检索并引用。

> 2025 年，用户搜索行为正加速向 AI 搜索迁移。研究表明，AI 搜索引擎的回答中**会明确引用信息来源**（Perplexity 显示引用链接，ChatGPT 搜索展示来源站点）。被 AI 引用 = 在新流量入口获得曝光。**GEO 是企业官网未来 2-3 年的新增长杠杆，目前几乎无建站产品覆盖。**

## 2. 目标（Goals）

### 2.1 用户目标

| # | 目标 | 衡量方式 |
|---|------|---------|
| U1 | 用户能在 **10 分钟内** 完成从"输入同行网址"到"得到可编辑的网站骨架" | 提交 URL 到模板生成完成时间中位数 ≤ 10 min |
| U2 | 客户 IT 人员能在 **30 分钟内** 完成本地部署并上线 | 从获取安装包到首站可访问的时间中位数 ≤ 30 min |
| U3 | 网站上线后，**搜索引擎收录与排名显著提升** | 上线 90 天内，核心关键词进入百度/Google 前 3 页比例 ≥ 60%；Google Search Console 收录页面数 ≥ 站点页面数 80% |
| U4 | 网站内容在 **AI 搜索引擎中被引用** | 上线 90 天内，在主流 AI 搜索（Perplexity/文心一言/豆包）针对行业问答的引用测试中，至少 1 个页面被引用 |
| U5 | 线索数据 100% 存储在客户本地 | 线索入库本地数据库成功率 100%，无任何数据外发 |

### 2.2 业务目标

| # | 目标 | 衡量方式 |
|---|------|---------|
| B1 | 验证"AI 结构复刻"为获客核心引擎 | 交付客户中，使用"参考同行"入口占比 ≥ 50% |
| B2 | 建立可规模化的 License 收入 | 上线 180 天内，累计交付客户 ≥ 100 家，年度服务费续费率 ≥ 70% |
| B3 | 以"SEO/GEO 效果"建立口碑壁垒 | 客户站点平均自然搜索流量 90 天内提升 ≥ 50%；NPS ≥ 40 |

## 3. 非目标（Non-Goals）

| # | 非目标 | 原因 |
|---|--------|------|
| N1 | **不做 SaaS 托管 / 多租户** | 明确走私有化路线，数据自主可控是核心卖点，与 SaaS 互斥 |
| N2 | **不做电商交易能力**（购物车/支付/订单） | v1 聚焦"官网展示 + 获客 + SEO/GEO"，电商属独立产品线 |
| N3 | **不做多语言/国际化站点** | v1 单一市场，多语言涉及翻译与 hreflang 路由，列 v2 |
| N4 | **不做 C 端会员系统** | v1 官网是"展示 + 获客"属性，不含 C 端账号体系 |
| N5 | **不做 SEO/GEO 的自动黑帽手段** | 不做关键词堆砌、隐藏文本、链接农场等违规手段，避免被搜索引擎惩罚 |
| N6 | **不直接抓取/复制原站文字、图片、品牌素材** | 合规红线，结构复刻仅复刻布局骨架，内容必须用户自行填写 |
| N7 | **不承诺具体排名名次** | 搜索排名受搜索引擎算法、行业竞争等多因素影响，产品提供优化能力与建议，不做排名名次承诺 |

## 4. 目标用户画像（Personas）

### P1 — 老王（企业主，建站决策与使用者）

- **画像**：38 岁，制造业小老板，公司 20 人，无专职技术团队，但有兼职 IT 或代运营可协助部署。
- **目标**：花钱少、上线快、能带来客户询盘，数据在自己手里。
- **痛点**：嫌 SaaS 建站长期订阅贵、数据不在自己手里；建好了没人搜得到。
- **成功标准**："软件装在自己服务器上，把同行网址丢进去就生成站，我把内容填好，百度搜我们行业能搜到，豆包问起来也能提到我们。"

### P2 — 小李（市场运营，建站执行者）

- **画像**：28 岁，市场专员，有一定审美和文案能力，负责建站与日常内容运营。
- **目标**：高效建站、持续优化 SEO/GEO、管理询盘线索。
- **痛点**：不懂技术 SEO；不知道怎么让 AI 搜索引用自家内容。
- **成功标准**："后台有 SEO/GEO 检查清单和优化建议，照着改就行，能看到收录和排名变化。"

### P3 — 阿强（IT/运维，部署与维护者）

- **画像**：30 岁，公司 IT 或外包运维，熟悉 Linux 基础操作，有 Docker 经验。
- **目标**：快速部署、稳定运行、方便备份升级。
- **痛点**：怕部署复杂、怕升级丢数据。
- **成功标准**："一条命令装好，有备份和升级脚本，出问题有文档和日志。"

### P4 — 张总（决策者，付费方）

- **目标**：性价比、效果可衡量、不被锁定。
- **决策依据**：一次买断价格透明、有 SEO/GEO 效果数据证明、数据自主。

## 5. 核心用户旅程（Core User Journey）

```
【一次性】本地部署
   │  IT 人员下载安装包 → 配置环境（Docker）→ 一键启动
   │  → 配置域名与 SSL → 系统初始化（管理员账号）
   │
【建站阶段】
   ├─→ 选择建站入口（三选一）
   │     ①【参考同行】输入同行官网 URL  ← 核心差异化入口
   │     ②【选行业模板】从内置模板库挑选
   │     ③【从空白开始】自由搭建
   │
   ├─→ 【入口①：AI 结构复刻链路】
   │     输入 URL → 爬取并解析页面结构 → 识别页面类型 → 提取区块骨架
   │       → 生成结构化模板（含占位内容，不含原站素材）→ 预览确认
   │
   ├─→ 【可视化编辑器】
   │     拖拽调整区块 → 替换文字/图片 → 调整主题色/字体 → 响应式预览
   │
   ├─→ 【SEO/GEO 配置】← 新增核心环节
   │     全局 SEO 设置 → 页面级 TDK → 结构化数据 → llms.txt 配置
   │       → GEO 内容优化建议 → 可引用性检查
   │
   ├─→ 【内容管理 CMS】
   │     发布产品/案例/文章 → 配置导航 → 内链结构优化
   │
   ├─→ 【表单自定义】
   │     设计询盘表单 → 投放页面 → 线索存入本地数据库
   │
   ├─→ 【发布】
   │     生成静态站点 → 部署到 Web 服务器 → 提交搜索引擎收录
   │
   └─→ 【持续运营】
         查看 SEO/GEO 仪表盘 → 监控收录/排名/AI 引用 → 持续优化内容
```

## 6. 用户故事（User Stories）

> 格式：作为 [角色]，我希望 [行为]，以便 [价值]。按优先级排序。

### 6.1 部署与系统管理

- **US-01**：作为 IT 人员，我希望通过一条 Docker Compose 命令完成系统部署，以便 30 分钟内让建站系统跑起来。
- **US-02**：作为 IT 人员，我希望有详细的部署文档和环境要求说明，以便我能独立完成部署。
- **US-03**：作为 IT 人员，我希望系统能自动备份和一键恢复，以便我不担心数据丢失。
- **US-04**：作为 IT 人员，我希望系统升级时不丢失已有网站数据与内容，以便平滑更新。
- **US-05**：作为企业主，我希望所有网站数据、线索、内容都存在我自己的服务器上，以便数据完全自主可控。

### 6.2 AI 结构复刻（核心）

- **US-10**：作为企业主，我希望输入一个同行官网链接，系统能自动生成结构相似的可编辑网站，以便不用从零设计快速起步。
- **US-11**：作为企业主，我希望解析过程显示进度反馈，以便知道系统在工作。
- **US-12**：作为企业主，我希望生成的模板使用占位文字和示例图片（而非原站内容），以便清楚需替换哪些内容且不涉及抄袭。
- **US-13**：作为企业主，我希望系统能识别原站包含哪些页面并自动创建对应骨架，以便网站结构完整。
- **US-14**：作为市场运营，我希望在解析结果基础上增删页面与区块，以便灵活调整结构。
- **US-15**：作为企业主，当网址无法解析时，我希望得到清晰原因和"改用模板"兜底建议，以便不卡住。

### 6.3 可视化编辑器

- **US-20**：作为企业主，我希望直接点击文字编辑，所见即所得，以便不用学操作。
- **US-21**：作为企业主，我希望拖拽调整区块顺序，以便快速调版面。
- **US-22**：作为企业主，我希望一键切换主题色和字体，以便快速统一风格。
- **US-23**：作为企业主，我希望切换 PC/平板/手机预览，以便确认移动端体验。
- **US-24**：作为市场运营，我希望编辑有撤销/重做和自动保存，以便不担心误操作。

### 6.4 内容发布系统（CMS）

- **US-30**：作为企业主，我希望在后台管理产品/案例/文章列表，以便持续丰富网站。
- **US-31**：作为企业主，我希望配置导航菜单和多级页面结构，以便组织信息架构。
- **US-32**：作为企业主，我希望一键发布和重新生成站点，以便掌控上线节奏。
- **US-33**：作为 IT 人员，我希望配置自己的域名和 SSL 证书，以便网站有正式访问地址。

### 6.5 SEO 搜索引擎优化（核心新增）

- **US-40**：作为市场运营，我希望系统自动生成规范的 sitemap.xml 和 robots.txt，以便搜索引擎高效抓取。
- **US-41**：作为市场运营，我希望为每个页面设置 SEO 标题/描述/关键词，以便优化搜索结果展示。
- **US-42**：作为市场运营，我希望系统自动生成结构化数据（Schema.org），以便搜索引擎理解我的内容（产品/文章/FAQ/面包屑）。
- **US-43**：作为市场运营，我希望系统给出 SEO 健康度检查清单和优化建议，以便我知道哪里没做好、怎么改。
- **US-44**：作为市场运营，我希望自动生成规范化的 URL slug 和 canonical 标签，以便避免重复内容影响收录。
- **US-45**：作为市场运营，我希望系统能自动向百度/Google/Bing 提交收录申请，以便加速收录。
- **US-46**：作为市场运营，我希望看到核心关键词的排名变化趋势，以便评估 SEO 效果。（P1）

### 6.6 GEO 生成式引擎优化（核心新增）

- **US-50**：作为市场运营，我希望系统生成 `llms.txt` 文件，以便 AI 搜索引擎高效理解并引用我的站点内容。
- **US-51**：作为市场运营，我希望系统对每篇内容做"可引用性评分"，告诉我内容是否容易被 AI 搜索引用，以便针对性优化。
- **US-52**：作为市场运营，我希望系统给出 GEO 优化建议（如增加事实陈述、数据点、引用来源、FAQ），以便提升 AI 引用概率。
- **US-53**：作为市场运营，我希望内容编辑器内置"AI 友好"写作辅助，提示我补充结构化信息（问答、定义、对比、统计数据），以便内容更易被 AI 提取。
- **US-54**：作为市场运营，我希望系统提供"AI 引用模拟测试"，输入问题查看我的内容是否可能被主流 AI 搜索引用，以便验证 GEO 效果。（P1）
- **US-55**：作为市场运营，我希望自动为内容添加权威性信号（作者署名、更新时间、引用来源标记），以便提升 AI 搜索的信任度。

### 6.7 表单自定义

- **US-60**：作为企业主，我希望拖拽字段设计询盘表单，以便收集需要的客户信息。
- **US-61**：作为企业主，我希望表单可放在联系页/弹窗/浮动按钮，以便多触点收集线索。
- **US-62**：作为企业主，我希望有人提交表单时收到邮件通知，以便及时跟进。
- **US-63**：作为企业主，我希望在后台查看线索列表并导出，以便管理复盘。线索全部存在本地。
- **US-64**：作为企业主，我希望支持微信/钉钉/飞书 Webhook 通知，以便团队协同跟进。（P1）

### 6.8 数据与运营

- **US-70**：作为企业主，我希望看到网站访问量、来源、热门页面，以便了解推广效果。
- **US-71**：作为企业主，我希望看到 SEO 仪表盘（收录数、排名、索引状态），以便监控搜索引擎表现。
- **US-72**：作为企业主，我希望看到 GEO 仪表盘（AI 引用模拟测试、可引用性评分分布），以便监控 AI 搜索表现。
- **US-73**：作为企业主，我希望访问统计数据支持接入自己的分析工具（如 Google Analytics / 百度统计），以便不依赖内置统计。

## 7. 功能需求（Requirements）

> 按 P0（Must）/ P1（Should）/ P2（Future）分级。每个需求含验收标准。

### 7.1 模块一：AI 网站结构解析引擎（核心差异化）

#### FR-1.1 同行 URL 输入与解析（P0）

**描述**：用户输入公开可访问的同行官网 URL，系统爬取目标站点，解析其页面类型与区块结构，生成结构化模板。

**验收标准**：
- [ ] 输入框支持 http/https URL，校验格式合法性
- [ ] 对无法访问/需登录/被反爬拦截的站点，返回明确提示并提供"改用行业模板"兜底

## 8. 指标体系（Success Metrics）

本地部署软件的指标逻辑与 SaaS 不同：**SaaS 看活跃与订阅，本地部署看交付成功率、客户成功与续费**。本产品核心价值是"建站 + 搜得到 + AI 引得到"，故北极星聚焦于**效果交付**。

### 8.1 北极星指标（North Star Metric）

> **NSM：客户站点搜索可见性指数（SVI, Search Visibility Index）**
>
> 定义：客户站点上线 90 天后，在传统搜索引擎（收录页面数 + 核心关键词排名分）与 AI 搜索引擎（内容可引用性评分 + AI 引用模拟通过率）的综合可见性得分。

**为什么是它**：本地部署 + License 模式下，客户付费的本质诉求是"建站能带来搜索流量与 AI 搜索曝光"。SVI 直接衡量产品是否兑现了"搜得到、AI 引得到"的核心承诺，是客户续费（年度服务费）与口碑传播的根本驱动。

| 周期 | 目标 |
|------|------|
| 客户上线 90 天 | SVI 较上线前提升 ≥ 50% |
| 客户上线 180 天 | 70% 客户 SVI 达到行业基准线以上 |

### 8.2 驱动指标（Leading）

| 漏斗环节 | 指标 | v1 目标 |
|----------|------|---------|
| 售前 → 交付 | 部署成功率（一次部署成功） | ≥ 95% |
| 部署 → 首站上线 | 部署到首站可访问时间 | 中位数 ≤ 30 min |
| 建站 → 使用"参考同行" | AI 复刻入口采用率 | ≥ 50% |
| 提交 URL → 生成模板 | 解析成功率 | ≥ 70% → 90% |
| 编辑 → 发布 | 发布转化率 | ≥ 60% |
| 发布 → SEO 生效 | Google/百度收录页面占比 | 90 天内 ≥ 80% |
| 发布 → GEO 生效 | AI 引用模拟通过率 | 90 天内 ≥ 40% |

### 8.3 健康指标（Lagging）

| 维度 | 指标 | 目标 |
|------|------|------|
| 续费 | 年度服务费续费率 | ≥ 70% |
| 满意度 | NPS | ≥ 40 |
| 服务质量 | 部署平均耗时 | ≤ 30 min |
| 稳定性 | 系统可用性（客户侧） | ≥ 99.5% |
| 效果 | 客户站点自然搜索流量 90 天提升 | ≥ 50% |
| 合规 | 复刻侵权举报数 | 0 起实质性侵权 |

### 8.4 护栏指标（Guardrail）

| 指标 | 含义 | 阈值 |
|------|------|------|
| 内容原创率 | 生成模板中照搬原站文案比例 | = 0% |
| 数据本地率 | 线索/内容外发到外部服务器的比例 | = 0%（除用户主动配置的第三方统计） |
| 白帽合规率 | 使用违规 SEO 手段的次数 | = 0 |

## 9. 风险与合规边界（Risk & Compliance）

### 9.1 ⚠️ 核心合规红线：结构复刻 ≠ 内容抄袭

| 维度 | 允许（结构复刻） | 禁止（内容抄袭） |
|------|------------------|------------------|
| 页面结构 | 复刻页面层级、导航栏目 | — |
| 区块布局 | 复刻区块类型与排列顺序 | — |
| 文字内容 | 使用系统占位文案 | 照搬原站标题、正文、slogan |
| 图片/视频 | 使用系统示例图 | 下载使用原站图片、Logo |
| 品牌元素 | — | 复刻原站 Logo、品牌色组合、商标 |
| 源码/样式 | — | 复制原站 HTML/CSS/JS |

**措施**：解析结果默认不携带原站文字/媒体；首次使用弹合规须知；后台对生成模板与原站做文本相似度监控。建议上线前法务 review。

### 9.2 SEO/GEO 合规

| 风险 | 应对 |
|------|------|
| 使用黑帽 SEO 被搜索引擎惩罚 | **严格白帽**，不做关键词堆砌/隐藏文本/链接农场；优化以内容质量与技术规范为主 |
| GEO 过度优化被 AI 平台判定为操纵 | GEO 以提升内容质量与可引用性为本，不做针对 AI 的虚假内容或操纵 |
| 排名效果不达预期引发客户纠纷 | **不做排名名次承诺**（N7），以"提供优化能力与建议"为产品边界，合同明确 |

### 9.3 爬取合规

| 风险 | 应对 |
|------|------|
| 违反目标站 robots.txt | 严格遵守，Disallow 路径不爬取 |
| 高频请求影响目标站 | 限频（单站 QPS ≤ 1），缓存结果 |
| 反爬拦截 | 优雅降级为"改用模板"，不绕过反爬 |
| 数据隐私 | 不存储原站用户数据，仅临时解析结构后丢弃原始 HTML |

### 9.4 本地部署特有风险

| 风险 | 等级 | 应对 |
|------|------|------|
| 部署环境差异导致失败 | 高 | Docker 标准化 + 部署验证脚本 + 详尽文档；提供最低配置要求 |
| 客户运维能力不足 | 中 | 提供托管运维增值服务；远程协助部署 |
| 数据备份缺失致丢数据 | 高 | 强制备份提醒 + 定时备份默认开启 + 升级前自动备份 |
| License 破解/盗版 | 中 | 机器码绑定 + 法律授权协议 + 源码授权模式溢价 |
| 升级兼容性 | 中 | 升级前自动备份 + 回滚能力 + 版本兼容性测试 |

### 9.5 技术风险

| 风险 | 等级 | 应对 |
|------|------|------|
| 解析成功率不达标 | 高 | 优先支持常见建站平台产物；SPA 用 Headless Browser 渲染；失败→模板兜底闭环 |
| GEO 效果难以量化验证 | 中 | 提供 AI 引用模拟测试作为参考，明确非绝对预测；持续跟踪主流 AI 平台行为变化 |
| AI 搜索算法变化 | 中 | GEO 策略以内容质量为本（不依赖特定技巧），持续迭代 |

## 11. 里程碑规划（Timeline Considerations）

### Phase 1 — MVP（M1-M3）：验证建站 + 部署 + 基础 SEO

**目标**：跑通"本地部署 → 参考同行 → 编辑 → 发布 → 基础 SEO"主链路。

**范围（P0）**：
- FR-9.1 / 9.2 / 9.3 / 9.4 部署、备份、升级、License
- FR-1.1 / 1.2 / 1.3 解析引擎
- FR-2.1 / 2.2 模板库（4 行业 × 3 套）
- FR-3.1 ~ 3.5 编辑器
- FR-4.1 ~ 4.4 CMS + 静态发布 + 域名 SSL
- FR-5.1 ~ 5.3 表单 + 本地线索
- FR-6.1 / 6.2 / 6.3 技术 SEO + Schema + 健康度检查
- FR-7.1 / 7.2 / 7.3 / 7.4 / 7.6 GEO 核心（llms.txt + 可引用性评分 + 写作辅助 + 权威信号 + 语义化）
- FR-8.1 / 8.4 访问统计 + 第三方接入

**成功标准**：10 家种子客户，≥ 80% 一次部署成功；≥ 50% 使用"参考同行"；上线 90 天收录率 ≥ 80%。

### Phase 2 — 增长（M4-M6）：补齐监控与变现

**目标**：SEO/GEO 效果可度量，建立续费与口碑。

**范围（P1）**：
- FR-1.4 多站点参考融合
- FR-5.4 通知扩展 + 防垃圾
- FR-6.4 / 6.5 收录提交 + 关键词排名监控
- FR-7.5 AI 引用模拟测试
- FR-8.2 / 8.3 SEO/GEO 仪表盘
- FR-9.5 源码授权模式

**成功标准**：续费率 ≥ 70%；客户站点自然流量 90 天提升 ≥ 50%；NPS ≥ 40。

### Phase 3 — 智能化（M7+）：建立壁垒

**目标**：GEO 能力持续领先，解析与内容生成智能化。

**范围（P2 + 新增）**：
- AI 辅助内容生成（根据行业自动填充 SEO/GEO 友好的示例文案）
- GEO 策略自适应（跟踪主流 AI 搜索算法变化）
- 模板智能推荐
- 多语言（v2）

## 附录 C：术语表

| 术语 | 含义 |
|------|------|
| 结构复刻 | 复刻目标网站页面结构、区块布局骨架，不含具体内容 |
