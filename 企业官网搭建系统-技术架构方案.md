## 1. 架构总览

### 1.1 架构原则

| 原则 | 说明 |
|------|------|
| **数据本地化** | 用户核心数据（线索、客户信息、媒体）100% 存本地；仅"解析 URL"和"GEO 内容分析（官网公开内容）"经用户授权走外网 |
| **静态发布** | 网站以纯静态 HTML 发布，编辑态与发布态分离，兼顾性能、安全与 SEO/GEO 友好 |
| **云端智能 + 本地执行** | 解析引擎的"智能"在我方云端持续迭代，本地负责编辑/发布/数据存储 |
| **容器化交付** | Docker Compose 一键部署，降低私有化交付门槛 |
| **GEO 双模式** | 规则引擎（内置零依赖兜底）+ 第三方 LLM API（用户自配 Key 增强）。无 Key 用规则跑基础功能，配 Key 调用大模型做智能分析；标准 2C4G 即可，无需高配服务器 |

### 1.3 数据流说明

| 数据流 | 方向 | 内容 | 合规说明 |
|--------|------|------|---------|
| 解析请求 | 本地 → 云端 | 同行官网 URL | URL 是公开数据，非客户敏感数据 |
| 解析结果 | 云端 → 本地 | 结构化 JSON（页面/区块骨架） | 不含原站文字/图片 |
| 网站内容/线索 | 仅本地 | 线索、客户信息、媒体等核心数据 | **零外发** |
| GEO 内容分析 | 本地 → 第三方 LLM API | 官网公开内容文本（用户授权） | **官网内容本就是即将公开发布的**，发给 API 分析不涉及隐私；线索/客户数据不外发；用户可随时关闭 |
| License 激活 | 本地 → 云端 | 机器码 + 订单号 | 用于签发 License，非业务数据 |
| 搜索引擎收录提交 | 本地 → 搜索引擎 | sitemap URL | 用户主动操作 |

## 2. 技术选型与决策依据

### 2.1 选型总览

| 层 | 技术 | 选型 | 关键理由 |
|----|------|------|---------|
| 前端（管理后台/编辑器） | React 18 + TypeScript + Vite | 所见即所得编辑器 | 生态成熟，dnd-kit 拖拽方案，团队招聘易 |
| UI 组件库 | shadcn/ui + Tailwind CSS | 管理界面 | 可定制、无运行时样式负担、设计感好 |
| 后端 API | Node.js + NestJS (TypeScript) | 全部业务 API | 与前端同语言降全栈成本；NestJS 企业级架构 |
| 数据库 | PostgreSQL 16 | CMS/线索/配置 | 关系型 + JSONB 灵活字段，成熟稳定 |
| 缓存/队列 | Redis 7 | 任务队列/缓存/会话 | 解析任务异步化、解析结果缓存 |
| SSG 框架 | **Astro** | 静态站点生成 | 内容站首选，零 JS 输出，SEO/GEO 极友好 |
| 解析渲染 | Playwright | Headless 浏览器 | 处理 SPA/动态渲染站点 |
| HTML 解析 | cheerio | DOM 提取 | 轻量快速 |
| 全文检索 | PostgreSQL FTS / BM25 | GEO 基础检索（无 API Key 时兜底） | 无需额外组件，内置即可 |
| LLM 能力 | **第三方 API 适配层** | GEO 智能分析（引用模拟/写作建议/内容优化） | 用户自配 Key，按需充值；千问/DeepSeek/智谱均兼容 OpenAI 格式 |
| Web 服务器 | Nginx | 反向代理 + 静态托管 | 行业标准，高性能 |
| 容器 | Docker Compose | 部署交付 | 一键启动，环境隔离 |
| License | RSA 签名 + 机器码 | 授权管理 | 离线可验证，防伪造 |

### 2.2 关键选型决策详述

#### 决策 1：为什么选 Astro 作为 SSG（解决 Q7）

| 候选 | 评价 | 结论 |
|------|------|------|
| **Astro** | 默认零 JS 输出（纯 HTML+CSS），Islands Architecture 按需 hydrate；内容站性能天花板；原生支持 Markdown/MDX；语义化 HTML 输出对 SEO/GEO 极友好；构建快 | **✅ 选定** |
| Next.js SSG | 功能全面但较重，企业官网不需要其 RSC/SSR 复杂度；输出含较多运行时 JS | ❌ 过重 |
| Hugo | 极快但 Go 模板学习成本高，且难以与 Node 编辑器深度集成 | ❌ 集成难 |
| Eleventy | 轻量但生态小，组件化能力弱 | ❌ 生态弱 |

**Astro 的 GEO 优势**：Astro 输出的 HTML 干净、语义化标签完整（`<article>`/`<section>`/`<nav>`/`<header>`/`<footer>`），这正是 AI 搜索引擎解析内容时最友好的结构。零 JS 意味着 AI 爬虫无需执行 JS 即可获取完整内容。

#### 决策 2：后端为什么选 NestJS 而非 Python

- **全栈 TypeScript**：前端编辑器与后端共享类型定义（如区块 schema、页面结构），减少接口对接成本。
- **I/O 密集型适配**：建站系统的后端主要是文件操作、数据库读写、HTTP 调度，Node.js 事件循环模型适合。
- **NestJS 企业级**：依赖注入、模块化、守卫/拦截器/管道，适合复杂业务逻辑的组织。
- Python 的优势在 ML，但本产品 P0/P1 的 GEO 不依赖重型 ML（见决策 4），无需 Python 后端。

## 3. 部署架构

### 3.1 Docker Compose 拓扑

```yaml
# docker-compose.yml 结构示意
services:
  nginx:        # 端口 80/443，反向代理 + 静态站点托管
  app:          # NestJS 后端 API + 管理后台静态资源
  ssg-worker:   # Astro 构建进程（消费构建队列）
  postgres:     # 数据持久化 (volume)
  redis:        # 队列/缓存 (volume)
```

```
                    ┌──────────────────────────────────┐
                    │        客户服务器 (Linux)         │
                    │        最低 2C4G / 推荐 4C8G      │
                    │                                   │
   :80/:443 ────────▶│  nginx                            │
                    │   ├── /api  ──▶ app (NestJS:3000) │
                    │   ├── /admin ──▶ app 静态资源      │
                    │   └── /     ──▶ /sites/<domain>   │
                    │                                   │
                    │  app ──▶ postgres:5432            │
                    │      ──▶ redis:6379               │
                    │      ──▶ ssg-worker (构建队列)     │
                    │                                   │
                    │  volumes:                         │
                    │   - pg_data (数据库)              │
                    │   - redis_data (缓存)             │
                    │   - media (用户上传)              │
                    │   - sites (SSG产物)               │
                    └──────────────────────────────────┘
```

### 3.2 部署流程

```
1. 下载安装包 siteforge-x.x.x.tar.gz
2. 解压 → 编辑 .env（域名、数据库密码、License Key）
3. docker compose up -d
4. 部署验证脚本自检（数据库连接 / Redis / Nginx / License 校验）
5. 访问 https://<域名>/admin → 初始化管理员账号
6. 开始建站
```

### 3.3 资源需求

| 配置 | 规格 | 适用场景 |
|------|------|---------|
| 最低 | 2C 4G 50G | 单站点，低流量 |
| 推荐 | 4C 8G 100G | 多站点，中等流量 |

> 注：GEO 智能分析通过第三方 LLM API 实现，**不占用本地算力**，标准 2C4G 即可运行全部功能。用户只需在后台配置 API Key（千问/DeepSeek/智谱）并自行充值，按调用量计费。无 Key 时降级为内置规则引擎，仍可使用基础 GEO 功能。

## 4. 核心子系统设计

### 4.1 AI 结构解析引擎（解决 Q2）

#### 决策：云端解析 + 本地兜底

**架构**：

```
本地编辑器                     云端解析服务
    │                              │
    │  ① 用户输入 URL              │
    │─────────────────────────────▶│
    │                              │ ② 检查 robots.txt
    │                              │ ③ Playwright 渲染页面
    │                              │ ④ cheerio 提取 DOM 结构
    │                              │ ⑤ 页面/区块分类模型识别
    │                              │ ⑥ 生成结构化 JSON
    │  ⑦ 返回结构骨架 JSON         │
    │◀─────────────────────────────│
    │                              │
    │  ⑧ 本地用占位内容填充模板     │
    │                              │
    │  ⑨ 云端不可用 → 本地兜底：    │
    │     引导选行业模板            │
```

**为什么是云端解析而非纯本地**：

| 维度 | 纯本地解析 | 云端解析（选定） |
|------|-----------|----------------|
| IP 封锁风险 | 客户服务器 IP 易被目标站封 | 云端可用代理池，降低封锁 |
| 模型迭代 | 需推送更新包，迭代慢 | 持续在线迭代，数据飞轮 |
| 性能影响 | 占用客户服务器资源 | 不影响客户服务器 |
| 数据隐私 | URL 不出本地 | URL 是公开数据，非敏感 |
| 离线可用 | ✅ | ❌（降级为模板兜底） |

**合规保障**：
- 云端仅接收 URL，返回结构 JSON，**不接触**客户的网站内容、线索、用户数据
- 严格遵守目标站 robots.txt
- 解析结果**不含原站文字/图片**，仅含结构骨架（页面类型 + 区块类型序列）
- 客户首次使用时明确告知"URL 解析由云端服务处理"

**解析引擎处理流程**：

```
输入 URL
  │
  ├─ robots.txt 检查 ──(Disallow)──▶ 拒绝 + 提示
  │
  ├─ Playwright 渲染（等待网络空闲，处理 SPA）
  │     └─ 超时 30s → 降级为纯 HTTP 抓取
  │
  ├─ cheerio 提取 DOM
  │     ├─ 导航栏 → 页面类型列表
  │     ├─ 主内容区 → 区块序列
  │     └─ 页脚 → 联系信息结构
  │
  ├─ 页面类型分类（URL 路径 + 导航文案 + 内容特征）
  │     └─ 首页 / 产品 / 案例 / 关于 / 联系 / 自定义
  │
  ├─ 区块类型分类（DOM 结构特征 + 视觉特征）
  │     └─ Hero / 图文 / 产品网格 / 案例 / 团队 / 统计 / FAQ / 表单 / 地图 / 页脚
  │
  ├─ 生成结构 JSON
  │     {
  │       pages: [{ type: "home", blocks: [{type:"hero"},{type:"feature_grid"}...] }, ...],
  │       navigation: [...]
  │     }
  │
  └─ 本地接收 → 用占位内容渲染为可编辑模板
```

**区块类型识别特征表**：

| 区块类型 | 识别特征 |
|----------|---------|
| Hero | 页面顶部大区域，含标题+副标题+CTA+背景图/视频 |
| 图文介绍 | 文本块 + 图片并排（左图右文/左文右图） |
| 产品网格 | 多个相似结构卡片网格排列，含图+标题+简介 |
| 案例列表 | 卡片/列表项含图片+标题+描述 |
| 团队介绍 | 圆形头像+姓名+职位排列 |
| 数据统计 | 大数字+标签的并排展示 |
| FAQ | 问答结构（`<details>` 或 折叠面板） |
| 表单 | `<form>` 元素含多个 input |
| 地图 | 嵌入地图 iframe/瓦片 |
| 页脚 | 页面底部多列链接区 |

### 4.2 静态站点生成引擎（解决 Q7）

#### 选型：Astro + 内容集合（Content Collections）

**工作原理**：

```
用户在编辑器操作
  │
  ├─ 内容保存 → PostgreSQL（结构化数据）
  │              + 本地 Markdown 文件（Astro Content Collections）
  │
  ├─ 点击"发布"
  │     │
  │     ├─ SSG Worker 消费构建队列
  │     ├─ Astro 读取 Content Collections + 页面配置
  │     ├─ 注入 SEO 标签（TDK / Schema / canonical / OG）
  │     ├─ 注入 GEO 标签（llms.txt / 语义化结构 / 权威信号）
  │     ├─ 生成纯静态 HTML/CSS/JS 到 /sites/<domain>/
  │     ├─ 生成 sitemap.xml / robots.txt / llms.txt
  │     └─ Nginx 刷新静态目录 → 站点更新上线
  │
  └─ 预览模式 → 构建到临时目录，不覆盖线上
```

**Astro Content Collections 映射**：

| CMS 内容类型 | Astro Collection | 路由 |
|-------------|------------------|------|
| 产品 | `products` | `/products/<slug>` |
| 案例 | `cases` | `/cases/<slug>` |
| 文章 | `posts` | `/posts/<slug>` |
| 页面 | `pages` | `/<slug>` |

**页面 = 区块序列**的渲染：

```astro
---
// 页面由区块序列组成，每个区块映射到一个 Astro 组件
import Hero from './blocks/Hero.astro'
import FeatureGrid from './blocks/FeatureGrid.astro'
import ProductList from './blocks/ProductList.astro'
import FAQ from './blocks/FAQ.astro'
import ContactForm from './blocks/ContactForm.astro'

const blockMap = { hero: Hero, feature_grid: FeatureGrid, ... }
const { page } = Astro.props
---

{page.blocks.map(block => {
  const Component = blockMap[block.type]
  return <Component {...block.props} />
})}
```

**关键设计：编辑器与 SSG 的区块 schema 统一**

编辑器中的"区块"与 Astro 组件共用同一份 TypeScript schema 定义，确保编辑器所见 = 发布所得：

```typescript
// shared/block-schema.ts（前后端共享）
export const blockSchema = z.discriminatedUnion('type', [
  z.object({ type: z.literal('hero'), props: heroPropsSchema }),
  z.object({ type: z.literal('feature_grid'), props: featureGridPropsSchema }),
  // ...每种区块一个定义
])
```

### 4.3 SEO 引擎

#### 架构

```
                    ┌─────────────────────────┐
                    │      SEO 引擎            │
                    │                          │
  内容编辑 ────────▶│  TDK 生成器              │──▶ <title>/<meta>/<og:>
                    │  Schema.org 注入器       │──▶ JSON-LD <script>
                    │  sitemap 生成器          │──▶ sitemap.xml
                    │  robots.txt 生成器       │──▶ robots.txt
                    │  URL slug 规范化         │──▶ canonical
                    │  健康度检查器            │──▶ 体检报告 + 评分
                    │  收录提交器              │──▶ 百度/Google/Bing API
                    └─────────────────────────┘
```

#### robots.txt 生成策略（SEO + GEO 双重考虑）


### 4.4 GEO 引擎（解决 Q4）

这是本产品最大的技术差异化。设计核心原则：**不让客户为 GEO 买单买硬件**。

#### 决策：规则引擎（内置兜底）+ 第三方 LLM API（用户自配增强）

> **为什么不本地部署 LLM**：本地跑 7B 量化模型需 16G+ 内存甚至 GPU，中小企业为建站系统配这种服务器不现实；且本地小模型能力远不如 DeepSeek-V3/Qwen-Max 等云端大模型。
>
> **为什么第三方 API 可行（数据合规）**：GEO 引擎分析的是**即将发布到互联网的官网内容**——这些内容本就是公开的，不是线索数据、不是客户信息。发给 API 分析不涉及隐私。需要留在本地的"线索/客户数据"依然 100% 在本地，两件事不冲突。

**双模式设计**：

| 模式 | 触发条件 | 能力范围 | 成本 |
|------|---------|---------|------|
| **规则引擎模式**（内置） | 未配置 API Key 时自动启用 | llms.txt 生成、可引用性评分（6 维度）、写作辅助（规则提示）、权威信号注入、语义化校验、BM25 基础检索 | 零成本，零依赖 |
| **LLM 增强模式** | 用户配置第三方 API Key | 上述全部 + AI 引用模拟测试、智能写作建议、内容优化生成、FAQ 自动生成草案 | 用户按 API 调用量自行充值，成本可控透明 |

**支持的服务商**（均兼容 OpenAI API 格式，统一适配）：

| 服务商 | 推荐模型 | 特点 |
|--------|---------|------|
| 阿里千问 | qwen-plus / qwen-max | 中文能力强，国内访问稳定 |
| DeepSeek | deepseek-chat | 性价比高，推理能力强 |
| 智谱 GLM | glm-4 / glm-4-flash | 国内合规友好 |
| OpenAI（可选） | gpt-4o-mini | 海外用户可选 |

#### (1) llms.txt 生成器（规则引擎，无 API 依赖）

基于 [llms.txt 规范](https://llmld.org/spec/llms-txt-v1)（Jeremy Howard 提出，Stripe/Cloudflare/Anthropic 等已采用），自动生成：

```markdown
# {站点名称}

> {站点一句话简介，自动从站点描述提取}

## 主要内容

- 产品服务: https://{domain}/products - {产品列表概述}
- 成功案例: https://{domain}/cases - {案例概述}
- 关于我们: https://{domain}/about - {企业简介概述}
- 联系方式: https://{domain}/contact - {联系方式}

## 核心资源

- {最新文章标题}: https://{domain}/posts/{slug}
- FAQ: https://{domain}/faq - {常见问题概述}

## 补充说明

{企业核心能力描述，2-3 句，帮助 AI 理解企业定位}
```

同时生成 `llms-full.txt`（详细版，含完整内容摘要）。**配了 API Key 时，可用 LLM 自动撰写"补充说明"段落，让 AI 更好理解企业定位。**

**生成时机**：每次站点发布时自动更新 `/llms.txt` 和 `/llms-full.txt`，用户可在后台手动编辑。

#### (2) 可引用性评分器（规则引擎，无 API 依赖）

对每篇内容从 6 个维度评分（0-100），逐项诊断：

| 维度 | 满分 | 检测逻辑 | 优化建议示例 |
|------|------|---------|-------------|
| 事实陈述密度 | 20 | 统计数字/百分比/具体数据出现频次 | "建议补充产品性能数据或客户案例数据" |
| 结构化程度 | 20 | FAQ/列表/表格/定义块占比 | "建议将此段转化为 Q&A 结构" |
| 引用来源 | 15 | 外链 + 来源标注数量 | "建议添加数据来源链接" |
| 权威性信号 | 15 | 作者署名 + 发布/更新时间 + 作者简介 | "建议补充作者信息与发布时间" |
| 内容完整度 | 15 | 字数 + 主题覆盖度 | "建议扩展至 800 字以上，覆盖核心问题" |
| 语义清晰度 | 15 | 标题层级 + 段落结构 + 关键词一致性 | "建议增加 H2 小标题组织内容" |

```typescript
// 可引用性评分（规则引擎，零依赖）
function scoreCitability(content: ParsedContent): ScoreResult {
  return {
    factDensity:    scoreFacts(content),        // 检测数字、百分比、统计
    structure:      scoreStructure(content),     // 检测 FAQ、列表、表格
    citations:      scoreCitations(content),     // 检测外链、来源标注
    authority:      scoreAuthority(content),     // 检测作者、时间戳
    completeness:   scoreCompleteness(content),  // 检测字数、覆盖度
    semanticClarity: scoreSemantic(content),     // 检测标题层级、段落
    total: sum(...),
    suggestions: generateSuggestions(...)
  }
}
```

> **LLM 增强可选**：配了 API Key 时，可让大模型对"语义清晰度"和"内容完整度"做更深入的主观判断，替代部分规则打分，评分更准。

#### (3) AI 友好写作辅助（编辑器内嵌）

**规则版（无 API Key）**——实时检测提示：

- 检测到产品描述无数据 → 💡 "补充性能数据可提升 AI 引用概率"
- 检测到长段落无小标题 → 💡 "添加 H2 小标题让 AI 更易提取"
- 检测到无 FAQ → 💡 "添加常见问答区块"
- 检测到无引用来源 → 💡 "标注数据来源增强权威性"
- 提供"GEO 写作模板"：行业问答模板、产品能力模板、解决方案模板

**LLM 增强版（有 API Key）**——更智能：

- 基于内容上下文生成具体改写建议（而非通用提示）
- 自动生成 FAQ 问题草案（基于内容提取潜在问题）
- 生成"缺失事实"的补充建议（如"建议补充 XX 行业的市场数据"）

#### (6) AI 引用模拟测试（LLM 增强功能，需 API Key）

> 这是 GEO 最有"感知"的功能，也是最能体现产品差异化的能力。仅在用户配置 API Key 后可用。

**核心思路**：AI 搜索引擎的引用逻辑本质是 RAG（检索增强生成）——先检索相关内容，再生成回答并引用来源。用第三方大模型模拟这个流程：

```
用户输入问题（如"哪家公司的 CRM 适合中小企业？"）
  │
  ├─ ① BM25 检索本站内容，召回 Top-K 相关片段（规则引擎，无 API）
  │
  ├─ ② 将召回片段 + 用户问题发给第三方 LLM：
  │     Prompt: "以下是某企业官网的内容片段。请基于这些内容回答用户问题。
  │              如果内容中有可直接引用的信息，请在回答中引用并标注来源。
  │              如果内容不足以回答，请说明缺失了什么。"
  │
  ├─ ③ LLM 返回：回答内容 + 是否引用了本站内容 + 引用了哪些片段
  │
  ├─ ④ 判断引用概率：
  │     高：LLM 引用了本站内容并生成完整回答
  │     中：LLM 部分引用，但指出信息不足
  │     低：LLM 表示内容与问题不相关
  │
  └─ ⑤ 输出报告：引用概率 + LLM 模拟回答 + 召回片段 + 优化建议
```

**实现要点**：
- 检索用内置 BM25（PostgreSQL FTS），零成本
- 生成判断调用第三方 LLM API，每次测试约消耗几百 token，成本极低
- 支持"优化前 vs 优化后"对比测试
- **明确告知用户**：此为模拟测试，非真实 AI 搜索结果预测；实际 AI 搜索结果受各平台算法影响

**API 调用成本估算**（让用户心里有数）：

| 操作 | Token 消耗 | 估算费用（DeepSeek） | 估算费用（千问） |
|------|-----------|---------------------|------------------|
| 单次引用模拟测试 | ~1000 token | ≈ ¥0.001 | ≈ ¥0.004 |
| 单篇内容智能优化建议 | ~800 token | ≈ ¥0.001 | ≈ ¥0.003 |
| FAQ 草案生成 | ~600 token | ≈ ¥0.001 | ≈ ¥0.002 |

> 一个企业官网每月做几十次 GEO 分析，月成本仅几元到十几元，远比配一台 16G 服务器划算。后台提供"用量统计"让用户掌控开销。

**表单提交处理（静态站点的特殊处理）**：

由于发布的是静态站点（无后端运行时），表单提交需要一个 API 端点。方案：

- 静态站点的表单 `POST` 到 `https://<domain>/api/leads/submit`
- Nginx 将 `/api/*` 反向代理到 NestJS 后端
- 后端验证 + 存储 + 触发通知
- **线索数据仅存本地 PostgreSQL，零外发**

### 4.8 License 授权系统（解决 Q3）

#### 决策：机器码绑定 + 离线激活 + RSA 签名

```
┌─────────────┐                    ┌─────────────────┐
│ 客户本地     │                    │ 我方 License 服务 │
│             │  ① 提交机器码+订单号  │                  │
│  机器码生成  │───────────────────▶│  License 签发    │
│  (硬件指纹)  │                    │  (RSA 私钥签名)   │
│             │  ② 返回 License 文件 │                  │
│             │◀───────────────────│                  │
│  License    │                    │                  │
│  验证       │                    │                  │
│  (RSA 公钥) │                    │                  │
│             │                    │                  │
│  ③ 离线运行  │                    │                  │
│  (定期校验   │                    │                  │
│   可选在线   │                    │                  │
│   续期)      │                    │                  │
└─────────────┘                    └─────────────────┘
```

**机器码生成**：

```typescript
// 采集硬件指纹生成唯一机器码
function generateMachineId(): string {
  const cpuId = readCpuId()        // /proc/cpuinfo
  const mac = getPrimaryMac()      // 主网卡 MAC
  const diskSerial = getDiskSerial() // 根分区磁盘序列号
  return sha256(`${cpuId}|${mac}|${diskSerial}`).substring(0, 32)
}
```

**License 文件结构**（JSON + RSA 签名）：

```json
{
  "licenseId": "SF-2025-00001",
  "machineId": "a3f8b2c1...",
  "domain": "www.example.com",
  "edition": "standard",
  "issuedAt": "2025-07-01",
  "expireAt": "2026-07-01",
  "features": ["seo", "geo", "forms", "cms"],
  "maxSites": 5,
  "signature": "RSA-SHA256 签名..."
}
```

**验证逻辑**：

1. 启动时读取本地 License 文件
2. 用内置 RSA 公钥验证签名
3. 校验 machineId 是否匹配当前机器
4. 校验 domain 是否匹配
5. 校验过期时间
6. **License 失效时**：已发布静态站点保持正常访问（Nginx 直接托管），仅管理/编辑功能锁定

**离线场景**：
- License 文件一次性导入，无需持续联网
- 可选配置"在线续期"，到期前提醒

### 5.2 JSONB 设计说明

区块 `props`、表单 `fields`、内容自定义字段等使用 JSONB 存储，原因：
- 灵活性：不同区块类型 props 不同，自定义字段用户可配置
- PostgreSQL JSONB 支持 GIN 索引，查询性能足够
- 与 Zod schema 配合做运行时校验

## 6. 安全设计

### 6.1 威胁模型与对策

| 威胁 | 对策 |
|------|------|
| 管理后台未授权访问 | JWT 认证 + 强密码策略 + 可选 2FA |
| 表单垃圾提交 | 频次限制 + 验证码(P1) + honeypot 字段 |
| SQL 注入 | ORM (Prisma) 参数化查询，禁止拼接 SQL |
| XSS | 富文本输出经 DOMPurify 过滤；CSP 头 |
| 文件上传恶意文件 | 类型白名单 + 魔数校验 + 重命名 + 隔离目录 |
| License 破解 | RSA 签名 + 机器码绑定 + 法律协议 |
| 静态站点篡改 | Nginx 只读挂载 sites 目录 |
| 备份数据泄露 | 备份文件加密 + 访问权限控制 |

### 6.2 数据隔离

- 单租户部署（每个客户独立实例），天然隔离
- 多站点场景：站点间数据通过 `site_id` 隔离，API 层强制校验

### 6.3 网络安全

- 强制 HTTPS（HTTP 自动 301）
- 管理后台可配置 IP 白名单
- 数据库/Redis 不暴露外部端口（仅 Docker 内网）

## 7. 阻塞项决策总结

| 阻塞项 | 决策 | 依据 |
|--------|------|------|
| **Q2 解析引擎自研/集成？是否联网？** | 云端解析服务（自研 Playwright+cheerio+分类模型）+ 本地模板兜底 | 解析需持续迭代（数据飞轮），客户 IP 易被封；URL 非敏感数据；本地保留兜底 |
| **Q3 License 激活方案？离线场景？** | 机器码绑定 + RSA 签名 License 文件 + 离线激活 | 满足离线部署；签名防伪造；失效时静态站保持可用 |
| **Q4 GEO AI 引用模拟如何实现？** | 双模式：规则引擎（内置零依赖兜底）+ 第三方 LLM API（用户自配 Key 增强，千问/DeepSeek/智谱） | 官网内容本就是公开的，发 API 分析无隐私风险；用户按需充值免高配服务器；标准 2C4G 即可运行 |
| **Q7 SSG 框架选型？** | Astro（Content Collections + 零 JS 输出） | 内容站首选，SEO/GEO 友好（语义化 HTML），性能极佳 |
| Q8 排名监控依赖第三方 API？ | 是，用户自行申请百度/Google 站长 API Key | 非核心，不阻塞 MVP |
